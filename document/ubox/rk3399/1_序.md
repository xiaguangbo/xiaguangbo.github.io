## 简介

rk3399 有 usb3.0 5Gbps otg，这点对于可以连接电脑的设备很好，传输数据的速度快  
用的是 firefly-rk3399 板卡

### 启动流程

rk3399 芯片内有 192KiB sram 和 32KiB rom  
启动时首先运行芯片内部预先存储的程序，该程序也称为 bootrom、maskrom，该程序从数个外部存储设备中找“ID Block”信息（查找顺序是 SPI Nor > SPI Nand > eMMC > SD/MMC/TF），找到了就认该设备为启动盘  
bootrom 加载 tpl 到芯片内部的 sram 并执行，tpl 会初始化 dram，结束后回到 bootrom，bootrom 再加载 spl 到 dram 并执行，此时 bootrom 就结束了。spl 会将 u-boot.itb 加载到 dram 中并执行

tpl 因为要初始化 dram 所以包含 dram 的驱动，这样 spl 就不需要驱动了。tpl 和 spl 都来自 uboot

### 启动盘里的数据分布

距离起点的位置：

```txt
sector(512Byte)   : byte                             : file
0x40     d64      : 0x8000     d3_2768      32 KiB : idbloader.img
0x4000   d1_6384  : 0x80_0000  d838_8608    8  MiB : u-boot.itb
0x8000   d3_2768  : 0x100_0000 d1677_7216   16 MiB : boot.img
0x40000  d26_2144 : 0x800_0000 d1_3421_7728 128MiB : rootfs.img
```

linux 里制作烧录镜像要先写好所有的分区表信息，再烧录文件。因为 linux 的 fdisk 会清空镜像的前 2048 个扇区，也就是 1MiB  
还需要 GPT分区表，但只有在 u-boot.itb（不包括u-boot.itb）后面的需要添加分区信息。分区信息只需位置和大小，分区里的文件系统由分立的镜像文件决定，比如 rootfs 分区的格式由 rootfs.img 本身的文件系统类型绝定  
镜像释放后表现为前 16MiB 是空闲，boot.img 里的文件在一个 fat16 分区（fat32 所在的空间不能低于 32MiB），rootfs.img 里的文件在一个 ext4 分区  
如果分区需要文件系统，比如 boot.img，就需要在制作 boot.img 时就将 boot.img 格式化为 fat16，并把对应的文件复制进去。之后就准备个大的 .img 文件，并写好分区表信息，再把各个镜像文件按照要求的位置写到大镜像文件里。rootfs.img 起点位置可以跟随在 boot.img 后面

- idbloader.img  
    主要包含 ID Block 包头、u-boot-tpl.bin、u-boot-spl.bin  
    这是 rk 芯片的私有格式  

- u-boot.itb  
    主要包含 u-boot-nodtd.bin、bl31.elf、u-boot.dtd，而旧式的 .img 不包含 bl31.elf，也就是 trust.img  
    bl31.elf 包含 TF-A 的 dram、sram、m0 的 .bin 文件。rk3399 内部 有两个 Cortex-M0，PMU M0 被 TF-A 使用，Perilp M0 给用户使用

- boot.img  
    主要包含内核镜像、内核用的设备树

- rootfs.img  
    主要包含用户使用的程序和文件

## 开发环境

在 windows docker 里的 archlinux 里进行开发  
安装 docker 时会提示要安装一个适用于 wsl 的包，手动安装后继续安装 docker

1. 创建并运行容器

    ```
    docker run -itd --privileged --name archlinux_container archlinux
    ```

    -itd：i：可交互。t：交互方式是终端。d：这次不进入容器。i 和 t 必须设置，不然容器无法运行  
    --privileged：可以有 root 权限。该容器默认是 root 用户，但是不加这个依然没有 root 权限  

2. 进入容器

    ```
    docker exec -it archlinux_container bash
    ```

    bash：进入容器后执行 bash，一个终端程序。这样才能正常使用终端

    也可以在 vscode 安装 docker 扩展，右击容器选择 Attach Shell 就能让容器的 shell 出现在 vscode 内嵌的终端上。而且可以使用该扩展直接编辑容器里的文件

3. 添加软件仓库

    在 /etc/pacman.d/mirrorlist 第一行添加：

    ```
    Server = https://mirrors.tuna.tsinghua.edu.cn/archlinux/$repo/os/$arch
    ```

4. 更新本地软件仓库信息并更新所有软件

    ```
    pacman -Sy
    pacman -Syu
    ```

    pacman -S 软件名: 安装软件  
    pacman -Ssq 关键字: 在仓库中搜索含关键字的软件包  
    pacman -Qs 关键字: 搜索已安装的软件包  
    pacman -Ru 软件名: 删除软件，同时删除不需要的依赖  

    搜索命令以获取包名：https://man.archlinux.org/man/rpcgen.1

5. 安装软件

    ```
    pacman -S vim sudo
    ```

6. 添加普通用户并修改密码

    ```
    useradd -m xiaguangbo
    passwd xiaguangbo
    ```

    用户组和用户名都是这 xiaguangbo

7. 让用户可以获得 root 权限

    ```
    vim /etc/sudoers
    ```

    找到 root ALL=(ALL:ALL) ALL，在下面添加 xiaguangbo ALL=(ALL:ALL) ALL，然后强制写入

8. 切换到普通用户

    ```
    su xiaguangbo
    ```

    ```
    mkdir ~/.bin
    wget https://storage.googleapis.com/git-repo-downloads/repo -P ~/.bin/
    ```

    这个 repo 是一个 python 文件，大小不超过 100KiB，不是本体
    
    ```
    chmod a+x ~/.bin/repo
    ```

3. 打开并移动到 ~/.bashrc 的末尾

    ```
    vim ~/.bashrc
    ```

4. 添加内容

    需要输入 i，进入编辑模式，然后调整光标位置再粘贴

##### 安装 yocto

1. 下载

    ```
    mkdir ~/ubox/yocto/firefly
    cd ~/ubox/yocto/firefly
    repo init -u https://gitlab.com/firefly-linux/manifests.git -b master -m yocto_release.xml
    ```

    init：初始化  
    -u：指定 manifests 仓库的地址。这是整个项目的清单  
    -b：指定 manifests 仓库的分支。将仓库 clone 下来后会切换分支  
    -m：指定 manifests 仓库中的清单文件
    
    yocto_release.xml 要注释掉报错的那行

2. 同步

    ```
    repo sync -c
    ```

    sync：同步。下载清单里的代码，并将本地代码更新到最新  
    -c：只同步指定的分支。默认情况下 sync 会同步所有的远程分支

##### 编译

1. 设置要编译的目标板

    ```
    cd ~/ubox/yocto/firefly/yocto
    source oe-init-build-env
    cd conf
    ln -sf rk3399/firefly-rk3399.conf local.conf
    ```

    将要编译的目标板的 xxx.conf 做一个 local.conf 链接放到 yocto/build/conf 下

2. 编译

    ```
    cd ~/ubox/yocto/firefly/yocto
    git config --global --add safe.directory '*'
    ```

    git config --global --add safe.directory '*'：有些文件的所有者有问题，如果所有者不对就会强制发生错误，这是设置所有的文件夹都是安全的

    ```
    source oe-init-build-env
    ```
    
    终端重启后就需要执行一遍

    会出现：

    ```
    ···
    core-image-minimal
    core-image-full-cmdline
    core-image-sato
    core-image-weston
    meta-toolchain
    meta-ide-support
    ···
    ```

    给 bitbake 用，这是选择编译出哪些东西，比如：

    ```
    bitbake core-image-minimal
    ```
  
    core-image-minimal：这是执行 source oe-init-build-env 之后出现的，从那里面选一个
    
    会下载和编译很多东西。如果某个包一直下载失败，就从报错出现的链接里下载下来并放到 yocto/packages 下，再复制任意一个 .done 文件改为对应名字，再找到其对应的 .bb 文件，比如在 yocto/meta-chromium/recipes-browser/chromium 下找到相关字样的 .bb 文件，然后在 .bb 文件里的 SRC_URI 里添加 file:///root/ubox/yocto/packages/chromium-96.0.4664.110.tar.xz \ 这样的，要具体到包名
