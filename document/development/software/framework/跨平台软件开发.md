人机交互部分和数据处理部分分离，也就是前后端分离，两者之间通过网络通信。前端是客户端，基于浏览器，后端是服务端，基于可创建服务器的软件。

程序分为两部分，加减乘除等逻辑和输入输出，最基本的就是单片机程序。程序下载到单片机里，单片机上电，程序开始运行，初始化一些外设，比如串口，用来接收和发送数据，使用串口发送数据给单片机，单片机再通过串口返回一些数据，或者就驱动个 led 什么的。单片机不能一直什么都不做，至少也要能输出，如果就像一个无底洞一样只进不出就没有意义了，。

所有的用户软件都依赖操作系统的 api，比如 qq、vscode，加减乘除虽然只和 cpu 有关，但是输入输出就不一样了，必须依赖操作系统提供的 api，不然软件去初始化电脑上的串口再用串口输入输出吗，或者去初始化 gpu、hdmi、dp、屏幕，然后再显示到屏幕上来，即使可以，那这个软件的画面又如何和其他软件的画面共存呢？显然不可能，而且也没必要这么做。现在主流的电脑、手机的操作系统就是 windows、linux、macos、android、ios，他们提供的输入输出的 api 不一样，一个软件用 windows 的 api，把软件放到 linux 上连安装都安装不了，因为安装器本身使用的 api 就不兼容 linux，如果要安装，那么就要换成 linux 的软件安装器，即使安装上了，软件主体用的某些 api 还是不兼容 linux，软件也就没法用。

程序语言的计算部分由 cpu 直接执行，输入输出则调用系统 api。由于系统 api 各不相同，所以就出现了兼容层，将不同的系统 api 换成一样的。比如 C 语言里的 printf 库函数，就需要为各个操作系统适配，这个 printf 最终要还是要调用系统提供的某个用于输出的函数。不同的程序语言无非是书写方式、内部数据存储排列不一样，至于是运行在单片机里还是运行在电脑、手机里就看有没有适配。

输入输出库要向编程语言提供统一 api，然后适配各操作系统以实现相同的功能。程序语言的计算指令需要针对 cpu 适配，而输入输出需要针对操作系统适配，那么这两者必然是可互相独立的。针对画面显示部分，操作系统只提供了非常原始的绘图方法，难道要软件开发者去使用这些垃圾 api？所以出现了 UI api 库，但是 UI 库进化非常快，功能也不再是原来直接操作 api，而是出现了框架，有各种辅助机制来辅助画面显示，比如 xml、css，所以各 UI 框架提供的 api 也不一样了。这种情况下，程序语言要么适配一个现有的 UI 框架，要么自己出一个配套的 UI 框架，这时如果使用的 UI 框架不能跨平台，那即使语言能跨平台又如何，这就很尴尬了。但是还有一种万能方法，软件整体的 UI 部分和计算部分分成两个独立的程序，UI 也有一个程序来控制，这个程序和软件的计算部分通过数据来交流，就是计算部分告诉 UI 控制程序显示什么数据，UI 控制程序告诉计算程序用户输入了什么，以实现输入输出的功能，这就是前后端分离。交流的数据总和平台无关吧，那么这个 UI 控制程序就可以不提供 api 给计算程序，UI 控制程序随便怎么升级，计算程序随便怎么更换 UI 控制程序，都可以做成同一个软件的样子，至于数据格式自定义就好。

目前适配多平台显示最好的就是浏览器，不论是电脑还是手机，都可以通过浏览器看到同一个网页，这也是因为浏览器适配了不同的操作系统。所以计算程序要跨平台显示，做成网页或者包装一个浏览器是不二之选，当然，浏览器的工作效率要差不少，毕竟要运行浏览器和 UI 框架。但浏览器对后端的要求就降低了很多，即使是单片机也可以使用网页来显示 UI，对于要本地显示的还是推荐适配了的 UI 框架。
